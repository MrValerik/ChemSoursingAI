# ChemSource AI

**ИИ-ассистент закупок химического сырья** — on-premise система для автоматизации сбора ценовых предложений и квалификации поставщиков.

Направление: агро и биотех · Итоговый продукт: on-premise ИИ-ассистент с веб-интерфейсом

---

## О проекте

Закупка химического сырья сегодня — трудоёмкий ручной процесс: специалист регистрирует запросы, ищет поставщиков и производителей, ведёт переписку с десятками контактов (преимущественно китайских) и вручную сводит ответы воедино. Ответы приходят в произвольном формате и часто без ключевых данных — цены, базиса поставки, спецификации (TDS) или сертификата анализа (CoA).

**ChemSource AI** принимает базовые параметры продукта (CAS-номер, наименование, чистота, применение, ценовой ориентир) и далее:

- проверяет вещество по CAS-номеру (PubChem);
- формирует стандартизированные RFQ под базисы CIP Moscow / FCA Shanghai / EXW;
- ведёт первичную переписку с поставщиками по email и WhatsApp;
- извлекает структурированные данные из ответов;
- сводит всё в сопоставимую таблицу для финального решения специалиста.

Система разворачивается полностью в контуре заказчика — содержимое переписки и коммерческие условия не передаются во внешние сервисы.

## Ключевые возможности

1. Приём входных данных (CAS, наименование, чистота, применение, история цен)
2. Верификация вещества по CAS и генерация RFQ (CIP/FCA/EXW)
3. Поиск поставщиков в открытых источниках
4. Авто-рассылка RFQ (Email / WhatsApp) с запросом цены, CoA, TDS
5. Извлечение данных из ответов и авто-дозапрос при неполноте
6. Сводная сравнительная таблица по поставщикам с экспортом (CSV/Excel)
7. Контроль полноты ключевых параметров
8. История коммуникаций (карточка менеджера)
9. Эскалация нестандартных кейсов на специалиста
10. Шаблоны ответов и ведение статусов запросов

## Архитектура

On-premise развёртывание (Docker Compose / Kubernetes). Блочная масштабируемая архитектура из семи логических слоёв со слабой связанностью и явными контрактами между модулями:

| Слой | Назначение |
| --- | --- |
| L1. Представление | SPA веб-интерфейс, BFF / API Gateway |
| L2. Прикладные сервисы | RFQ, Поставщики, Котировки, Коммуникации, Эскалации, Шаблоны, Auth/RBAC |
| L3. Оркестрация | Оркестратор ИИ-конвейера (workflow-движок) |
| L4. ИИ/ML | LLM-инференс, извлечение, OCR, перевод, эмбеддинги, Retrieval/RAG |
| L5. Интеграции | Коннекторы: Email, WhatsApp, Echemi, веб-сорсинг |
| L6. Асинхронность | Брокер сообщений, очереди, пул воркеров |
| L7. Данные и наблюдаемость | PostgreSQL + pgvector, объектное хранилище, кэш, логи/метрики/трейсы, Eval |

### ИИ-конвейер обработки запроса

`Верификация (CAS / PubChem)` → `Генерация RFQ` → `Сорсинг` → `Рассылка` → `Извлечение (structured output)` → `Контроль полноты + авто-дозапрос` → `Сводка и эскалация`

## Технологический стек

| Компонент | Технология |
| --- | --- |
| Frontend | React + TypeScript, Vite |
| Backend | Python (FastAPI) либо Node.js (NestJS) |
| ИИ / парсинг (MVP) | Qwen3-8B (Q4_K_M, GGUF) через встроенный llama-server (llama.cpp) на RTX 4060 8 ГБ; OCR — PaddleOCR/Tesseract на CPU; эмбеддинги bge-m3 на CPU |
| ИИ / парсинг (прод) | Qwen3-32B / 30B-A3B + Qwen2.5-VL через vLLM |
| База данных | PostgreSQL + pgvector |
| Очередь / воркеры | Redis / RabbitMQ + Celery |
| Email | IMAP/SMTP-коннектор (полностью в контуре) |
| WhatsApp | Cloud API (см. примечание по приватности) |
| Инфраструктура | Docker Compose / Kubernetes, GPU-узел для LLM |

Локальная LLM встроена в приложение: бэкенд запускает `llama-server` как дочерний процесс и обращается к нему по HTTP через OpenAI-совместимый эндпоинт. Стороннего сервиса (Ollama и т.п.) ставить не нужно. Смена модели при масштабировании — изменение конфигурации, а не кода.

## Подход к качеству извлечения

Гибрид правил и LLM с человеком в петле (HITL):

- **structured output** — LLM с жёсткой JSON-схемой котировки (function calling), а не свободный текст;
- **детерминированные валидаторы** — CAS по контрольной сумме, цены/валюты/единицы по regex и справочникам, Incoterms по словарю;
- **оценка уверенности по каждому полю** — поля с низким confidence уходят на ручную проверку или авто-дозапрос;
- **golden-датасет** — 100–300 размеченных реальных писем для измерения точности (precision/recall);
- **RAG по истории закупок** — ценовые ориентиры подтягиваются из pgvector.

## Целевые метрики

| Цель | MVP | Продакшен |
| --- | --- | --- |
| Время от запроса до сводной таблицы | — | ≤ 3 дней |
| Доля RFQ по единому шаблону | 100% | 100% |
| Доля котировок без ручного ввода | ≥ 70% | ≥ 85% |
| Полнота котировок (цена, базис, MOQ, спецификация) | ≥ 80% | ≥ 90% |
| Точность полей цена/базис/MOQ | ≥ 90% | ≥ 95% |

## Безопасность и приватность

Решение работает в контуре заказчика: ролевой доступ (RBAC), журнал действий, шифрование вложений (CoA/TDS) и контролируемый исходящий шлюз с белым списком доменов.

> **Примечание по WhatsApp.** После отключения On-Premises API (Meta, 23.10.2025) единственный путь — WhatsApp Cloud API, при котором сообщения проходят через инфраструктуру Meta. Это противоречит принципу «данные не покидают контур», поэтому режим обработки WhatsApp-данных согласуется с заказчиком. Email-канал (IMAP/SMTP) может быть полностью в контуре и предпочтителен для конфиденциальных условий.

## Этапы реализации

Итерации по 2 недели, команда 4–5 человек. Ориентировочно до пилота — **12–16 недель**.

0. Подготовка — доступы, baseline-метрики, развёртывание Qwen3-8B
1. MVP-прототип — веб-интерфейс, имитация конвейера
2. Ядро + БД — backend, модель данных, генерация RFQ, сводная таблица
3. ИИ-парсинг — локальная LLM, OCR, извлечение, golden-датасет
4. Интеграции — Email, WhatsApp Cloud API, веб-сорсинг, PubChem
5. Эскалация и история — карточки менеджеров, шаблоны, статусы
6. Пилот и приёмка — on-premise развёртывание, обучение, доработки
