# Структура репозитория

Раскладка соответствует блочной архитектуре из ТЗ (7 слоёв). Каждый каталог — зона ответственности с явными границами.

```
ChemSoursingAI/
├── backend/                  # L1–L3, L7: API, прикладные сервисы, оркестрация, данные
│   └── app/
│       ├── api/              # L1: HTTP-эндпоинты (BFF/API Gateway)
│       ├── core/             # конфиг, настройки, общие утилиты
│       ├── models/           # L7: ORM-модели (сущности из ТЗ)
│       ├── schemas/          # Pydantic-схемы запросов/ответов и JSON-схема котировки
│       ├── services/         # L2: бизнес-логика (RFQ, котировки, эскалации, шаблоны)
│       ├── connectors/       # L5: адаптеры внешних интеграций (PubChem, Email, WhatsApp)
│       └── workers/          # L6: асинхронные задачи (рассылка, парсинг, дозапрос)
├── frontend/                 # L1: SPA — рабочее место специалиста (React + TS)
├── ml/                       # L4: ИИ/ML
│   ├── extraction/           # конвейер извлечения (structured output + валидаторы)
│   ├── prompts/              # промпты и JSON-схемы
│   └── golden_dataset/       # размеченные письма для замера precision/recall
├── data/
│   └── synthetic/            # синтетический корпус писем для отладки без боевых доступов
├── deploy/                   # docker-compose / k8s, конфигурация развёртывания
└── docs/                     # ТЗ, план реализации, документация
```

## Принципы

- **Слабая связанность.** Внешние интеграции скрыты за слоем `connectors` с единым интерфейсом и fallback — смена провайдера не затрагивает ядро.
- **LLM через OpenAI-совместимый API.** Смена модели (llama-server → vLLM) — изменение конфигурации, а не кода.
- **Конфигурация извне.** Все параметры — через переменные окружения (см. `.env.example`).
- **Онлайн/офлайн разделение.** Тяжёлый парсинг и индексация выполняются воркерами асинхронно, не конкурируя с интерактивным слоем.
